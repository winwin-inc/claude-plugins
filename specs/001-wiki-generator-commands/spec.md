# 功能规范：Claude Code Wiki 生成命令

**版本**: 1.0.0
**创建日期**: 2025-01-03
**状态**: 草稿

## 1. 功能概述

### 1.1 简短名称
`wiki-generator-commands`

### 1.2 功能描述
创建一个简化的 Claude Code 自定义命令 `/wiki-generate`，用于自动分析和生成项目 Wiki 文档。该命令将利用 AI 的代码分析能力，为项目自动生成、更新和维护高质量的技术文档。通过参数控制行为，在保持简单性的同时提供足够的灵活性。

### 1.3 业务价值
- **减少 70% 的文档维护工作量**：通过 AI 自动分析代码生成文档
- **保持文档与代码同步**：代码变更时自动更新相关文档
- **标准化文档结构**：所有项目文档遵循统一格式和质量标准
- **降低新人上手成本**：自动生成清晰的架构和模块文档
- **提升文档质量**：AI 生成专业技术内容，包含代码示例和图表

## Clarifications

### Session 2025-01-03

- Q1: 命令粒度和用户交互模式 - 应该使用 19 个独立命令还是单一命令？
  - A1: 单一命令 `/wiki-generate`，通过参数控制行为。简单、易记、满足 90% 使用场景。

- Q2: 单一命令 `/wiki-generate` 需要支持哪些参数？是否需要包含所有 19 个原始命令的功能？
  - A2: 核心功能（5个参数）：
    - `--full`：完整生成（初始化 + 概览 + 架构 + 所有模块 + API + 开发指南 + 索引）
    - `--update`：增量更新（基于 Git 提交分析受影响的文档）
    - `--module=path`：单个模块文档生成
    - `--validate`：质量检查和评分
    - `--translate=lang`：翻译到其他语言
  - 满足 90% 使用场景，保持最简单易用。

- Q3: 增量更新时如何保护用户的手动编辑内容？需要平衡保护性和易用性。
  - A3: **智能检测跳过策略**：
    - AI 检测用户手动修改的文档章节（通过 Git diff 分析）
    - 更新时跳过已手动修改的章节，保留用户内容
    - 其他未修改的章节正常更新
    - 在变更报告中列出跳过和更新的章节
  - **优势**：最小化用户干预，自动保护手动编辑，平衡灵活性和保护性

## 2. 用户场景与测试

### 2.1 主要用户角色

**角色 1：项目维护者**
- **需求**：保持项目文档与代码同步
- **痛点**：手动更新文档耗时且容易遗漏
- **目标**：代码变更后自动更新文档

**角色 2：新加入开发者**
- **需求**：快速理解项目架构和模块
- **痛点**：文档过时或不完整
- **目标**：通过生成的文档快速上手

**角色 3：技术文档编写者**
- **需求**：生成标准化文档模板
- **痛点**：从零开始编写文档结构费时
- **目标**：AI 生成初稿，人工优化

### 2.2 核心用户流程

#### 流程 1：新项目初始化文档

**场景**：开发者创建新项目，需要生成完整的 Wiki 文档

**步骤**：
1. 开发者在项目根目录执行 `/wiki-generate --full`
2. AI 自动检测项目结构并初始化文档目录
3. 生成项目概览、架构文档、模块文档、API 文档、开发指南
4. 生成导航索引
5. 自动验证文档质量

**预期结果**：
- `docs/` 目录包含完整的文档结构
- 所有文档质量分数 ≥ 80
- 文档包含代码示例和图表
- 一条命令完成所有操作

#### 流程 2：增量更新文档

**场景**：开发者修改了某个模块的代码，需要更新对应文档

**步骤**：
1. 开发者完成代码修改并提交
2. 执行 `/wiki-generate --update`
3. AI 分析最近的 Git 提交，识别受影响的模块（如 `src/auth`）
4. 自动更新对应的模块文档
5. 生成变更报告

**预期结果**：
- 只更新受影响的文档
- 保留手动编辑的内容
- 变更报告清晰列出修改内容

#### 流程 3：单个模块文档生成

**场景**：开发者添加了新模块 `src/payment`，需要生成文档

**步骤**：
1. 开发者执行 `/wiki-generate --module src/payment`
2. AI 分析 `src/payment/` 目录
3. 生成 `docs/modules/payment.md`
4. 包含：模块概述、API 接口、依赖关系、使用示例

**预期结果**：
- 文档准确反映模块功能
- 包含可运行的代码示例
- 列出所有依赖关系

#### 流程 4：文档质量验证

**场景**：发布前确保文档质量达标

**步骤**：
1. 执行 `/wiki-generate --validate`
2. AI 检查所有文档
3. 生成质量报告

**预期结果**：
- 质量分数 0-100
- 列出优点和需要改进的地方
- 如果分数 < 80，提供具体改进建议

#### 流程 5：多语言文档导出

**场景**：需要为国际团队提供英文文档

**步骤**：
1. 执行 `/wiki-generate --translate en`
2. AI 翻译所有文档为英文
3. 生成 `docs/en/` 目录

**预期结果**：
- 保持文档结构和格式
- 技术术语准确翻译
- 代码示例不翻译

## 3. 功能需求

### 3.1 核心命令

#### 命令：`/wiki-generate [参数]` - Wiki 文档生成器

**描述**：统一的 Wiki 文档生成命令，通过参数控制生成行为

**命令格式**：
```bash
/wiki-generate --full
/wiki-generate --update
/wiki-generate --module=<path>
/wiki-generate --validate
/wiki-generate --translate=<lang>
```

---

#### 参数 1：`--full` - 完整生成

**描述**：生成完整的项目 Wiki 文档（首次使用或全面重建）

**功能**：
1. **初始化阶段**：
   - 创建 `docs/` 目录结构
   - 创建 `docs/modules/` 子目录
   - 创建 `docs/api/` 子目录
   - 创建 `docs/diagrams/` 子目录
   - 生成 `.claude/wiki-config.json` 配置文件

2. **分析阶段**：
   - 扫描项目根目录（README、package.json、requirements.txt 等）
   - 识别项目类型和技术栈
   - 分析项目结构和目录组织
   - 识别所有模块和组件

3. **生成阶段**：
   - 生成项目概览文档（`docs/00-Overview.md`）
   - 生成架构文档（`docs/01-Architecture.md`）
   - 为所有模块生成文档（`docs/modules/*.md`）
   - 生成 API 文档（`docs/api/ENDPOINTS.md`）
   - 生成开发指南（`docs/02-Development.md`）
   - 生成导航索引（`docs/README.md`）

4. **优化阶段**：
   - 生成 Mermaid 图表
   - 验证文档质量
   - 生成统计报告

**输出**：
- 完整的 `docs/` 目录结构
- 控制台进度消息
- 最终质量报告

**验收标准**：
- 目录创建成功
- 所有核心文档生成完整
- 质量分数 ≥ 80
- 执行时间 < 5 分钟（中型项目）

**性能**：< 5 分钟（100 个模块的项目）

---

#### 参数 2：`--update` - 增量更新

**描述**：基于最近的代码提交，智能更新受影响的文档

**功能**：
1. **变更检测**：
   - 分析最近 5 次 Git 提交
   - 识别修改的文件路径
   - 识别新增和删除的文件

2. **影响分析**：
   - 映射修改的文件到受影响的模块
   - 确定需要更新的文档列表
   - 检测依赖关系变化

3. **智能更新**（核心特性）：
   - **检测手动修改**：通过 Git diff 分析文档，识别用户手动编辑的章节
   - **选择性更新**：
     - 跳过已手动修改的章节（保留用户内容）
     - 更新未修改的章节（同步代码变化）
   - 更新受影响的模块文档、API 文档、架构图
   - **100% 保证不丢失手动编辑内容**

4. **变更报告**：
   - 生成变更摘要
   - 列出跳过的章节（手动编辑保护）
   - 列出更新的章节（同步代码变化）
   - 追加到 `docs/CHANGELOG.md`

**输出**：
- 更新的文档文件
- `docs/CHANGELOG.md`（追加）
- 控制台变更报告
  ```
  📝 变更报告
  ✅ 更新：docs/modules/auth.md
     - 跳过：'使用示例'（检测到手动修改）
     - 更新：'API 接口'（代码变化同步）
  ```

**验收标准**：
- 只更新受影响的文档
- 保留手动编辑内容（100% 保证，通过 Git diff 检测）
- 变更报告清晰完整（列出跳过和更新的章节）
- 不删除现有内容（只增强）

**性能**：< 2 分钟（增量更新）

**实现机制**：
- 使用 Git diff 比较当前文档与上次 AI 生成版本
- 识别手动编辑：检测文档章节的内容变化
- 更新策略：保留手动章节，重写其他章节

---

#### 参数 3：`--module=<path>` - 单模块文档生成

**描述**：为指定模块生成详细文档

**输入**：
- `path`：模块路径（如 `src/auth`、`lib/utils`、`app/components/Header`）

**功能**：
1. **模块分析**：
   - 扫描指定目录下的所有代码文件
   - 提取导出的函数、类、接口
   - 识别模块的主要功能和职责
   - 分析 import/require 依赖关系

2. **文档生成**：
   - 生成模块概述
   - 列出所有 API 接口（函数签名、参数、返回值）
   - 绘制依赖关系图
   - 生成可运行的使用示例
   - 提取 JSDoc/DocString 注释

3. **文档输出**：
   - 文件：`docs/modules/<module-name>.md`
   - 包含：概述、API 列表、依赖关系、使用示例、注意事项

**输出**：
- 文档：`docs/modules/<module-name>.md`
- 控制台生成消息

**验收标准**：
- API 接口覆盖率 ≥ 80%
- 每个接口有代码示例
- 依赖关系准确
- 文档包含实际代码片段

**性能**：< 10 秒（单个模块）

---

#### 参数 4：`--validate` - 文档质量验证

**描述**：全面检查文档质量并给出评分

**功能**：
1. **结构检查**：
   - 验证文档完整性
   - 检查标题层级结构
   - 验证内部链接有效性

2. **内容检查**：
   - 检查代码示例准确性
   - 评估内容完整性
   - 验证 Mermaid 图表语法

3. **质量评分**：
   - 规则引擎评分（60%）：结构、链接、代码示例等
   - AI 评分（40%）：内容质量、可读性、专业性
   - 总分范围：0-100

4. **生成报告**：
   - 列出优点（通过的检查项）
   - 列出需要改进的地方（未通过的检查项）
   - 提供具体改进建议

**输出**：
- 控制台：质量报告
  ```
  📊 Wiki 文档质量报告
  总分：85/100

  ✅ 优点：
  - 文档结构清晰完整
  - 代码示例丰富

  ⚠️ 需要改进：
  - [01-Architecture.md] 缺少部署架构图
  - [03-API.md] 部分链接失效
  ```

**验收标准**：
- 质量分数计算准确
- 报告列出具体问题
- 提供改进建议
- 检查时间 < 30 秒

**性能**：< 30 秒（全部文档）

---

#### 参数 5：`--translate=<lang>` - 翻译文档

**描述**：将文档翻译为其他语言

**输入**：
- `lang`：目标语言代码（en、ja、ko、fr、de 等）

**功能**：
1. **语言检测**：
   - 识别源文档语言
   - 验证目标语言支持

2. **智能翻译**：
   - 翻译所有文档
   - 保持 Markdown 格式和结构
   - **保持代码示例不翻译**
   - **保持技术术语不翻译**（如 HTTP、API、SQL）
   - 保持 Mermaid 图表代码不翻译

3. **目录组织**：
   - 创建 `docs/<lang>/` 目录
   - 保持与原文档相同的文件结构
   - 复制图片和图表

**输出**：
- 目录：`docs/<lang>/`（如 `docs/en/`）
- 所有翻译后的文档
- 控制台翻译进度和统计

**验收标准**：
- 翻译准确率 ≥ 85%
- Markdown 格式完全保持
- 代码示例不被翻译
- 技术术语保留原文
- 链接正确指向翻译版本

**性能**：< 3 分钟（全部文档）

### 3.2 参数组合使用

某些参数可以组合使用以实现更复杂的需求：

**示例 1：生成完整文档并验证**
```bash
/wiki-generate --full --validate
```
先完整生成文档，然后自动运行质量检查。

**示例 2：更新模块文档并验证**
```bash
/wiki-generate --module=src/auth --validate
```
生成单个模块文档后验证质量。

**示例 3：更新并翻译**
```bash
/wiki-generate --update --translate=en
```
先增量更新文档，然后翻译为英文。

### 3.3 配置管理

#### 配置文件：`.claude/wiki-config.json`

**功能**：存储 Wiki 生成器的配置

**配置项**：

```json
{
  "output_dir": "docs",
  "exclude_patterns": [
    "node_modules",
    "dist",
    "build",
    ".git",
    "coverage"
  ],
  "template_dir": ".claude/templates",
  "quality_threshold": 80,
  "diagrams": {
    "enabled": true,
    "detail_level": "medium"
  },
  "modules": {
    "auto_detect": true,
    "patterns": ["src/*", "lib/*", "app/*"]
  },
  "language": "zh-CN"
}
```

**配置说明**：
- `output_dir`：文档输出目录
- `exclude_patterns`：排除的目录模式
- `template_dir`：自定义模板目录
- `quality_threshold`：质量分数阈值
- `diagrams.enabled`：是否生成图表
- `diagrams.detail_level`：图表详细级别（low/medium/high）
- `modules.auto_detect`：是否自动检测模块
- `modules.patterns`：模块目录匹配模式
- `language`：默认文档语言

**验收标准**：
- 配置文件在 `--full` 参数首次运行时生成
- 配置可被用户修改
- 命令读取配置并应用

### 3.5 错误处理

#### 常见错误场景

**错误 1：目录不存在**
```bash
/wiki-generate --module=src/auth
# 错误：模块路径 src/auth 不存在
```
**处理**：提示用户检查路径，建议使用 `/wiki-generate --full` 初始化

**错误 2：不是 Git 仓库**
```bash
/wiki-generate --update
# 错误：当前目录不是 Git 仓库
```
**处理**：提示 `--update` 需要 Git 仓库，建议使用 `--full` 生成完整文档

**错误 3：配置文件损坏**
```bash
# 错误：.claude/wiki-config.json 格式错误
```
**处理**：备份旧配置，生成默认配置，提示用户重新配置

**错误 4：质量分数低于阈值**
```bash
/wiki-generate --validate
# 警告：文档质量分数 75，低于阈值 80
```
**处理**：显示详细改进建议，不阻止命令执行

## 5. 关键实体

### 5.1 文档实体

**属性**：
